# Быстрый Эксперимент: первый сравнимый прогон ROME / MEMIT / LoRA

## Summary

Цель первого этапа: как можно быстрее получить **реальные сравнимые результаты** по `ROME`, `MEMIT`, `LoRA` на локальной `Qwen2.5-3B` без расширения исследования до `SFT`, `LocFT-BF` и последовательного `ROME`.

Выбранный минимальный сценарий:
- использовать уже доступный локальный `PubTator`
- брать текущий тип фактов `entity -> PMID`
- собрать **малый clean-набор 10–20 кейсов**
- прогнать на нем `ROME`, `MEMIT`, `LoRA`
- сохранить результаты по каждому кейсу и агрегированную сводку

Это не финальный научный benchmark, а **быстрый рабочий baseline**, который подтверждает пайплайн и дает первые числа для сравнения.

## Key Changes

### 1. Зафиксировать минимальный датасет для первого прогона
Собрать один небольшой `jsonl`-набор через уже существующий `build_pubtator_edits.py`:
- источник: локальный `PubTator`, лучше начать с `gene`
- размер: целиться в `10–20` edit-cases после фильтрации
- фильтрация оставить строгой:
  - `max_objects_per_subject = 1`
  - `max_subjects_per_object = 1`
  - `inverse_max_entities_per_pmid = 1`
- сохранить набор как стабильный первый benchmark для повторяемых прогонов

Результат:
- один фиксированный файл `my_exp/data/edits/...jsonl`
- один metadata-файл с параметрами выборки и статистикой фильтрации

### 2. Сделать batch-runner по edit JSONL
На базе текущего `my_exp/scripts/test_qwen25_3b_methods.py` сделать новый runner для пакетного прогона:
- вход:
  - путь к `jsonl`
  - список методов: `LoRA`, `ROME`, `MEMIT`
  - лимит по числу кейсов
  - директория результатов
- поведение:
  - читать кейсы построчно
  - для каждого `case_id` запускать отдельный edit
  - сохранять raw metrics по каждому методу и кейсу
  - не падать на первом неудачном кейсе; ошибки писать в результат
- выход:
  - per-case JSON results
  - один агрегированный summary JSON/CSV

Минимально поддерживаемые поля из dataset:
- `case_id`
- `prompt`
- `subject`
- `ground_truth`
- `target_new`
- `rephrase_prompt` или первый из `rephrase_prompts`
- `locality_prompt`
- `locality_ground_truth`
- `portability_prompt`
- `portability_ground_truth`

### 3. Сделать простую агрегацию метрик
Добавить максимально простую итоговую сводку:
- число успешно обработанных кейсов
- число ошибок по каждому методу
- средний `rewrite` success
- средний `rephrase` success
- средний `portability` success
- средний `locality` preservation
- среднее время на кейс
- при наличии: пиковая GPU memory или хотя бы стартовая/конечная оценка

Если структура raw metrics у разных методов немного отличается:
- нормализовать сводку в один плоский формат
- если какой-то ключ отсутствует, писать `null`, а не ломать расчет

### 4. Ограничить первый этап по объему
Чтобы не раздувать работу, **не включать** в первый быстрый прогон:
- `FT`
- `KN`
- `SFT` на множестве примеров
- `LocFT-BF`
- `ROME` с последовательным редактированием
- графики и большой отчет

Вместо этого оставить только то, что нужно для первого результата:
- dataset
- batch-runner
- summary
- короткая интерпретация результатов

## Public Interfaces / Inputs

Новые или обновленные интерфейсы должны быть минимальными:

### Batch runner CLI
Пример целевого интерфейса:
```bash
conda run -n EasyEdit python my_exp/scripts/run_edit_benchmark.py \
  --dataset my_exp/data/edits/pubtator_gene_first_benchmark.jsonl \
  --methods LoRA ROME MEMIT \
  --max-cases 15 \
  --output-dir my_exp/results/qwen25-3b/pubtator_gene_first
```

### Dataset builder CLI
Оставить текущий CLI как источник benchmark-набора; при необходимости только использовать его в документации/командах:
```bash
conda run -n EasyEdit python my_exp/data_processing/build_pubtator_edits.py ...
```

### Result format
Итоговый каталог должен содержать:
- raw results по кейсам
- один `summary.json`
- при желании один компактный `summary.csv`

## Test Plan

### Smoke checks
- builder создает `jsonl` и metadata на малом sample
- batch-runner читает `jsonl` без ручных правок
- каждый из методов `LoRA`, `ROME`, `MEMIT` запускается хотя бы на `1–2` кейсах

### Functional checks
- прогон на `10–20` кейсах завершается без полного обвала пайплайна
- при падении отдельного кейса runner продолжает остальные
- summary корректно считает только завершенные кейсы
- raw results позволяют вручную сопоставить `case_id -> method -> metrics/error`

### Acceptance criteria
Первый этап считается завершенным, если:
- есть фиксированный benchmark-файл на `10–20` кейсов
- есть один командный запуск для `LoRA/ROME/MEMIT`
- есть агрегированная сводка по методам
- можно кратко ответить:
  - какой метод чаще успешен на `rewrite`
  - какой лучше держит `locality`
  - какой чаще падает или дольше работает

## Assumptions

- Используем текущую модель `Qwen2.5-3B`, потому что она уже локально подготовлена и реалистична для быстрого цикла.
- Для первого результата берем `PubTator entity -> PMID`, хотя это упрощенный и неидеальный тип факта.
- Первый набор делаем маленьким и чистым, а не большим и шумным.
- На этом этапе важнее **рабочий сравнимый baseline**, чем исследовательская полнота.
- Следующий этап после этого плана: либо расширять benchmark до richer biomedical facts, либо добавлять `SFT/LocFT-BF/seq ROME` уже поверх работающего batch pipeline.
