# Plan v2: следующий этап после baseline

## Зачем

`plan.md` закрыл первый быстрый этап:

- есть рабочий benchmark
- есть batch-runner
- есть raw results
- есть summary и markdown report

Теперь задача не в том, чтобы еще раз гонять тот же baseline, а в том, чтобы сделать следующий шаг к содержательному исследованию.

## Главная цель v2

Получить **второй, более полезный экспериментальный контур**, который:

- остается достаточно простым и быстрым
- сохраняет текущий рабочий pipeline
- улучшает интерпретируемость результатов
- приближает benchmark к реальному biomedical knowledge editing

## Что делаем в v2

### 1. Зафиксировать baseline v1

Текущий прогон `pubtator_gene_first` считаем опорной точкой.

Что нужно:

- не перезаписывать его
- хранить как baseline `v1`
- использовать его как точку сравнения для всех следующих прогонов

Минимальный результат:

- текущие файлы остаются как есть
- новые эксперименты идут в новые каталоги результатов

### 2. Добавить улучшенную аналитику поверх текущего benchmark

Сейчас есть:

- `rewrite`
- `rephrase`
- `portability`
- `locality`
- `time`
- `peak gpu`

Нужно добавить:

- composite score, например:
  - `rewrite * locality`
  - `0.5 * rewrite + 0.5 * locality`
- автоматическое выделение:
  - лучших кейсов
  - худших кейсов
  - кейсов с самым сильным trade-off
- отдельную таблицу:
  - качество
  - устойчивость
  - цена по времени
  - цена по памяти

Минимальный результат:

- report generator умеет строить расширенный markdown report
- в отчете есть не только средние значения, но и interpretation-friendly блоки

### 3. Подготовить второй benchmark на другом подмножестве

Текущий baseline сделан на `gene -> PMID`.

Следующий практичный шаг:

- не менять весь pipeline
- собрать еще один небольшой benchmark:
  - либо `disease`
  - либо `mutation`
- прогнать те же `LoRA / ROME / MEMIT`
- сравнить с `gene`-версией

Зачем:

- понять, сохраняется ли тот же паттерн методов
- проверить, не был ли первый результат слишком специфичен к `gene`

Минимальный результат:

- новый `jsonl` benchmark
- новый каталог результатов
- новый markdown report
- короткое сравнение `gene` vs `disease` или `gene` vs `mutation`

### 4. Подготовить переход к более содержательным biomedical facts

Текущий тип факта:

- `entity mentioned in PMID`

Это удобно для быстрого старта, но слабо как исследовательский knowledge fact.

Поэтому в v2 не обязательно сразу реализовывать новый relation dataset, но обязательно нужно подготовить для этого основу:

- зафиксировать, какой тип richer facts нужен дальше
- определить, откуда его можно взять
- понять, как он будет маппиться в текущий `edit jsonl` формат

Минимальный результат:

- короткая техническая заметка или план-конвертер
- список целевых relation types
- понимание, что именно будет benchmark v3

## Порядок работы

### Шаг 1

Доработать `generate_benchmark_report.py`:

- composite metrics
- best / worst cases
- таблица trade-off
- краткие auto findings

### Шаг 2

Собрать второй benchmark:

- начать с `disease`, если он читается так же просто из PubTator
- если там окажется мало clean кейсов, перейти на `mutation`

### Шаг 3

Прогнать:

- `LoRA`
- `ROME`
- `MEMIT`

на втором benchmark и получить:

- raw results
- summary
- report

### Шаг 4

Сделать короткое сравнение двух baseline-наборов:

- где `LoRA` устойчивее
- где `ROME/MEMIT` лучше сохраняют locality
- как меняются время и память

### Шаг 5

После этого решить, что делать в v3:

- richer biomedical relations
- `SFT / LocFT-BF`
- sequential `ROME`

## Что не делаем в v2

Чтобы не расползтись по объему, в v2 пока не включаем:

- полноценный новый relation extraction pipeline
- массовую переработку PubTator в сложные графы отношений
- `SFT`
- `LocFT-BF`
- sequential `ROME`
- большие визуализации и графики

## Артефакты v2

К концу этапа должны появиться:

- улучшенный markdown report generator
- второй benchmark dataset
- второй benchmark result directory
- сравнение двух baseline-наборов
- короткая заметка о переходе к richer biomedical facts

## Критерий завершения

v2 считается завершенным, если:

- текущий baseline не сломан
- есть второй benchmark на другом типе сущностей
- есть сравнимые результаты для `LoRA / ROME / MEMIT`
- есть более сильный отчет, чем просто `summary.json`
- понятно, что именно будет следующим содержательным исследовательским шагом
