# Контекст проекта

Проект посвящён экспериментам по редактированию знаний малой языковой модели в нефтегазовой предметной области.

Основная задача — проверить, можно ли добавлять в модель новые доменные факты без полного переобучения и при этом контролировать:

* усвоение нового факта;
* обобщение на разные формулировки вопроса;
* сохранение ранее внесённых фактов;
* сохранение общих недоменных знаний модели;
* сохранение близких доменных знаний.

В качестве домена выбрана нефтегазовая тематика. Для экспериментов подготовлены 20 редактируемых фактов в формате триплетов `subject -> relation -> object`.

Пример:

```text
Кероген -> является источником -> углеводородов
Гидроразрыв пласта -> создаёт -> трещины в продуктивном пласте
Проппант -> удерживает -> трещины гидроразрыва открытыми
```

# Описание данных

## `data/triplets/oilgas.json`

Содержит 20 основных нефтегазовых фактов в формате триплетов.

Используется как базовый список знаний, которые будут редактироваться.

Поля:

```text
fact_id — идентификатор факта
subject — сущность
relation — отношение
object — целевое значение
level — служебная метка группы факта
```

## `data/edit_requests/oilgas_edit_requests.json`

Содержит входные данные для методов knowledge editing.

Используется для ROME, MEMIT и EasyEdit-подобных пайплайнов.

Поля:

```text
fact_id — связь с триплетом
prompt — вопрос для редактирования
subject — редактируемая сущность
target_new — новый целевой ответ
target_old — старый ответ, если известен
```

## `data/train_sets/lora_train.json`

Минимальный обучающий набор для LoRA SFT.

На каждый факт есть одна QA-пара:

```text
input -> output
```

Этот файл нужен как простой baseline: что будет, если не использовать специальный KE-метод, а просто дообучить LoRA на целевых фактах.

## `data/train_sets/lora_augmented_train.json`

Расширенный обучающий набор для LoRA / LocFT-BF с аугментацией.

На каждый факт добавлены дополнительные формулировки вопроса. Этот файл нужен, чтобы проверить, помогает ли обучение на парафразах лучше переносить внесённое знание на разные запросы.

## `data/evaluate_set/fact_questions.json`

Основной набор проверочных вопросов вокруг каждого редактируемого факта.

Содержит типы вопросов:

```text
direct — прямой вопрос по факту
paraphrase — переформулировка прямого вопроса
reverse — обратный вопрос
neighbor — вопрос по близкому доменному знанию
locality — вопрос для проверки локальности
```

Этот файл используется в обоих экспериментах: одиночном и последовательном редактировании.

## `data/evaluate_set/domain_questions.json`

Общий нефтегазовый eval-набор.

Он нужен, чтобы измерить, насколько модель знает нефтегазовую область до и после редактирования.

Этот набор не используется для обучения.

## `data/evaluate_set/general_questions.json`

Общий недоменный locality-набор.

Содержит вопросы по общим знаниям: география, история, литература, биология, математика, программирование и базовая наука.

Он нужен, чтобы проверить, не испортились ли обычные знания модели после редактирования нефтегазовых фактов.

## `data/sequential/sequential_order.json`

Фиксирует порядок внесения 20 фактов в последовательном эксперименте.

Это нужно, чтобы эксперимент был воспроизводимым и не зависел от порядка строк в `oilgas.json`.

Дальше проект движется в сторону реализации экспериментального пайплайна.

Порядок работ:

```text

3. Реализовать одиночное редактирование.
4. Реализовать последовательное редактирование.
5. Посчитать метрики.

7. Построить таблицы и графики.
```

# Baseline evaluation

Перед любым редактированием нужно прогнать исходную модель по всем оценочным наборам.

Прогоняются:

```text
fact_questions.json
domain_questions.json
general_questions.json
```

Цель baseline:

```text
1. Понять, какие нефтегазовые факты модель уже знает.
2. Понять, какие факты модель не знает или знает нестабильно.
3. Зафиксировать исходное качество по доменной области.
4. Зафиксировать исходное качество по общим знаниям.
```

Результаты baseline сохраняются в:

```text
outputs/baseline/
```

# Эксперимент 1: одиночное редактирование

Одиночное редактирование проверяет, как метод работает на одном факте без накопления изменений.

Схема:

```text
1. Берём исходную модель.
2. Редактируем один факт.
3. Прогоняем evaluation.
4. Сохраняем результат.
5. Откатываемся к исходной модели.
6. Берём следующий факт.
7. Повторяем для всех 20 фактов.
```

То есть каждый факт редактируется независимо.

Цель эксперимента:

```text
Понять, какой метод лучше всего вносит один нефтегазовый факт и меньше всего нарушает остальные знания модели.
```

В этом эксперименте оцениваются:

```text
direct questions
paraphrase questions
reverse questions
neighbor questions
locality questions
domain questions
general questions
```

Результаты сохраняются в:

```text
outputs/single_edit/
```

# Эксперимент 2: последовательное редактирование

Последовательное редактирование проверяет накопление правок.

Схема:

```text
исходная модель
+ oilgas_001 -> evaluation
+ oilgas_002 -> evaluation
+ oilgas_003 -> evaluation
...
+ oilgas_020 -> final evaluation
```

В отличие от одиночного эксперимента, модель не откатывается после каждой правки. Каждая следующая правка вносится поверх предыдущих.

Цель эксперимента:

```text
Проверить, может ли модель последовательно усваивать нефтегазовые факты без стирания предыдущих правок и без деградации общих знаний.
```

После каждого шага проверяются:

```text
1. Новый внесённый факт.
2. Все ранее внесённые факты.
3. Общий нефтегазовый domain-набор.
4. Общий недоменный locality-набор.
```

Результаты сохраняются в:

```text
outputs/sequential_edit/
```

# Метрики

## Reliability

Показывает, усвоила ли модель целевой факт.

Считается по прямым вопросам:

```text
Reliability = доля правильных ответов на direct questions
```

## Generalization

Показывает, переносится ли внесённый факт на другие формулировки.

Считается по парафразам:

```text
Generalization = доля правильных ответов на paraphrase questions
```

## Reverse score

Показывает, работает ли знание в обратной постановке вопроса.

Считается по reverse questions.

## Neighbor score

Показывает, не испортились ли близкие доменные знания вокруг редактируемого факта.

Считается по neighbor questions.

## Locality

Показывает, не сломались ли знания, которые не должны были измениться.

Есть два вида locality:

```text
fact-level locality — вопросы из fact_questions.json с типом locality
global locality — вопросы из general_questions.json
```

## Domain score

Показывает общее качество модели в нефтегазовой области.

Считается по:

```text
domain_questions.json
```

Используется для сравнения качества до и после редактирования.

## Retention

Используется только в последовательном эксперименте.

Показывает, сохраняются ли ранее внесённые факты после следующих правок.

```text
Retention = доля ранее внесённых фактов, которые модель продолжает правильно воспроизводить
```

# Итоговая оценка

Для общей оценки не используем простое среднее арифметическое, потому что оно может завышать результат.

Например, если факт не усвоился, но locality осталась высокой, среднее может выглядеть нормально, хотя метод фактически провалился.

Поэтому используем гармоническое среднее.

Для одиночного редактирования:

```text
EditQuality = HarmonicMean(
  Reliability,
  Generalization,
  Locality
)
```

Дополнительно можно смотреть `Reverse score` и `Neighbor score` отдельно.

Для последовательного редактирования:

```text
SequentialQuality = HarmonicMean(
  Reliability,
  Generalization,
  Locality,
  Retention
)
```

# Проверка правильности ответа

Ответ модели оценивается не только через точное совпадение строки.

Используется несколько уровней проверки:

```text
1. Нормализация текста.
2. Exact match.
3. Alias match.
4. При необходимости — semantic similarity или ручная проверка спорных случаев.
```

Для каждого вопроса в eval-наборах есть:

```text
expected_answer
aliases
```

Это нужно, потому что модель может ответить правильно, но другой формулировкой.

Пример:

```text
expected_answer: "углеводороды"
aliases: ["углеводородов", "нефть и газ", "углеводородное сырьё"]
```

# Какие методы сравниваем

Минимальный набор методов:

```text
1. LoRA SFT baseline
2. LoRA SFT с аугментацией
3. ROME
4. MEMIT
5. LocFT-BF
6. LocFT-BF с аугментацией
```

При необходимости часть методов можно отложить, но базовая логика сравнения такая:

```text
LoRA показывает эффект обычного дообучения.
ROME показывает точечное редактирование одного факта.
MEMIT показывает редактирование группы фактов.
LocFT-BF проверяет идею локализованного LoRA-based редактирования.
Аугментация показывает, помогает ли обучение на парафразах.
```

# Главный смысл практической части

Практическая часть должна ответить на вопрос:

```text
Какие методы позволяют малой языковой модели постепенно усваивать редкие нефтегазовые факты без полного переобучения и без существенного разрушения уже имеющихся знаний?
```

Одиночное редактирование показывает качество изолированной правки.

Последовательное редактирование показывает, насколько метод подходит для непрерывного накопления доменных знаний.

Если метод хорошо работает на одном факте, но плохо сохраняет предыдущие правки, он не подходит для непрерывного редактирования. Если метод чуть хуже вносит одиночный факт, но лучше сохраняет locality и retention, он может оказаться более полезным для реального сценария доменной адаптации.
