# Контекст проекта

`my_exp_2` — это текущий исследовательский контур проекта по `knowledge editing` для доменной терминологии на базе `EasyEdit`.

В этом контуре проверяется, можно ли вносить новые нефтегазовые факты в модель `Qwen2.5-3B` без полного переобучения и при этом контролировать качество усвоения, обобщения и сохранности остальных знаний.

## Главная исследовательская цель

Проверить, насколько методы редактирования знаний подходят для сценария доменной адаптации терминологии, когда:

- нужно добавить или скорректировать узкоспециализированный факт;
- нельзя полноценно переобучать всю модель;
- важно не испортить уже существующие знания;
- важно понимать, как модель ведет себя не только на одном edit prompt, но и на связанных вопросах.

Сейчас доменом выбрана нефтегазовая тематика.

## Что именно сравнивается

В текущей версии сравниваются три метода:

- `LoRA`
- `ROME`
- `MEMIT`

Сравнение идет в двух режимах:

1. `single-edit`
2. `sequential-edit`

## Почему именно эти два режима

### Single-edit

Показывает, как метод работает на одном факте в изоляции.

Это полезно, чтобы понять:

- умеет ли метод вообще вносить нужный факт;
- насколько хорошо он переносит правку на переформулировки;
- ломает ли он близкие знания вокруг edited fact.

### Sequential-edit

Показывает, как метод ведет себя при серии правок поверх уже измененной модели.

Это важно для реального сценария доменной адаптации, где знания добавляются не по одному разу на пустую базу, а постепенно.

Здесь уже важно не только качество текущей правки, но и:

- сохранение ранее внесенных фактов;
- накопительная деградация;
- устойчивость на общих и доменных вопросах после нескольких шагов.

## Текущий датасет

В качестве рабочего domain benchmark используется небольшой нефтегазовый набор из 20 фактов.

Формат базового факта:

```text
subject -> relation -> object
```

Примеры:

```text
Кероген -> является источником -> углеводородов
Гидроразрыв пласта -> создаёт -> трещины в продуктивном пласте
Проппант -> удерживает -> трещины гидроразрыва открытыми
```

Для каждого факта подготовлены:

- edit request;
- прямые вопросы;
- парафразы;
- обратные вопросы;
- соседние доменные вопросы;
- locality-вопросы.

Отдельно есть:

- общий нефтегазовый eval-набор;
- общий недоменный eval-набор;
- фиксированный порядок последовательного редактирования.

## Что в проекте уже реализовано

На данный момент это уже не черновой план, а рабочий пайплайн.

Реализовано:

- загрузка и валидация oilgas JSON-данных;
- baseline evaluation исходной модели;
- одиночное редактирование на `LoRA`, `ROME`, `MEMIT`;
- последовательное редактирование на `LoRA`, `ROME`, `MEMIT`;
- target-old resolution перед редактированием;
- агрегация метрик в JSON и CSV;
- markdown-отчеты по single-edit и sequential-edit.

## Как устроен baseline

Перед любым редактированием прогоняется исходная модель по:

- `fact_questions.json`
- `domain_questions.json`
- `general_questions.json`

Цель baseline:

1. понять, что модель уже знает;
2. увидеть шумные и нестабильные ответы;
3. зафиксировать исходное качество по домену;
4. зафиксировать исходное качество по общим знаниям.

## Как устроен single-edit

В `single-edit` каждый факт редактируется независимо.

Схема:

```text
base model
-> edit one fact
-> evaluate
-> save result
-> discard edited state
-> reload base model
```

Такой режим нужен для чистого сравнения edit methods без накопления прошлых правок.

## Как устроен sequential-edit

В `sequential-edit` правки накапливаются:

```text
base model
+ fact_1 -> evaluate
+ fact_2 -> evaluate
+ fact_3 -> evaluate
...
```

После каждого шага оценивается:

- текущий edited fact;
- ранее внесенные факты;
- общие недоменные вопросы;
- общие доменные вопросы.

## Что означают основные метрики

### Для single-edit

- `reliability` — усвоение прямого edit fact;
- `generalization` — перенос на парафразы;
- `reverse` — корректность на обратных вопросах;
- `neighbor` — поведение на близких доменных вопросах;
- `fact_locality` — локальность вокруг edited fact;
- `global_locality` — сохранность общих знаний;
- `domain_score` — сохранность общих нефтегазовых знаний;
- `edit_quality` — компактная сводная оценка.

### Для sequential-edit

- `current_reliability` — качество текущего шага;
- `current_generalization` — перенос текущего шага;
- `retention` — сохранение ранее внесенных фактов;
- `global_locality` — сохранение общих знаний после накопления правок;
- `domain_score` — сохранение доменных знаний после накопления правок;
- `sequential_quality` — сводная последовательная оценка.

## Логика target_old

Для некоторых методов важно не только новое значение `target_new`, но и текущее значение `target_old`.

В проекте используется отдельная логика разрешения `target_old`:

- перед одиночной правкой — на исходной модели;
- перед каждым sequential step — на текущем состоянии модели.

Если модель отвечает пусто, шумно или нестабильно, происходит fallback к `target_new`. Это позволяет трактовать кейс как добавление нового знания, а не как замену уже существующего.

## Практический смысл текущей версии

Текущая версия проекта уже позволяет получить воспроизводимое сравнение трех методов knowledge editing в нефтегазовом домене.

Она подходит для:

- первых сравнительных прогонов;
- анализа одиночных и последовательных правок;
- подготовки отчетов и обсуждения результатов;
- дальнейшего расширения до SFT, LocFT-BF и более сложной оценки.
