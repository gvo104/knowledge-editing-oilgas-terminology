# Контекст my_exp_2

`my_exp_2` — текущий исследовательский контур проекта по `knowledge editing` для доменной терминологии на базе `EasyEdit`.

Цель: проверить, можно ли локально добавлять или корректировать нефтегазовые факты в `Qwen2.5-3B` без полного переобучения и при этом контролировать усвоение факта, перенос на другие формулировки, локальность и сохранение ранее внесенных знаний.

## Текущий статус

Полный pipeline уже реализован и выполнен.

Актуальные результаты лежат здесь:

```text
my_exp_2/outputs/oilgas_qwen25_3b_full_20facts
```

В полном запуске сравнивались четыре метода:

- `LoRA`
- `ROME`
- `MEMIT`
- `WISE`

Режимы:

- `baseline`;
- `single-edit` на 20 фактах;
- `sequential-edit` на 20 фактах;
- полный `sequential eval` по всем уже внесенным фактам, full general/domain и generation details.

## Данные

Домен: нефтегазовая терминология.

Основной формат факта:

```text
subject -> relation -> object
```

Примеры:

```text
Кероген -> является источником -> углеводородов
Гидроразрыв пласта -> создаёт -> трещины в продуктивном пласте
Проппант -> удерживает -> трещины гидроразрыва открытыми
```

Размеры:

- 20 редактируемых фактов;
- 260 fact questions;
- 40 domain questions;
- 50 general questions;
- 20 sequential steps.

Для каждого факта есть direct/paraphrase/reverse/neighbor/locality вопросы. Поля `subject`, `relation` и `level` используются для дальнейшей аналитики.

## Baseline

Исходная `Qwen2.5-3B`:

| Набор | Accuracy |
|---|---:|
| `fact_questions` | `0.053846` |
| `domain_questions` | `0.025` |
| `general_questions` | `0.68` |

Вывод: модель почти не знает подготовленный нефтегазовый factual benchmark, но сохраняет приемлемое качество на общих недоменных вопросах.

## Метрики оценки

В проекте используется два типа оценки:

- `EasyEdit internal metrics`: метрики, которые возвращает сам `EasyEdit` после редактирования;
- `generation-based metrics`: собственная проверка через генерацию ответа модели и сравнение с `expected_answer` / `aliases`.

Для generation-based оценки ответ нормализуется:

- приводится к нижнему регистру;
- `ё` заменяется на `е`;
- удаляется лишняя пунктуация;
- убираются служебные role-токены и шумные хвосты;
- затем проверяется, содержит ли ответ ожидаемый ответ или один из aliases.

Если совпадение найдено, вопрос считается решенным. Accuracy считается как:

```text
accuracy = число правильных ответов / число всех вопросов
```

### Идейная механика расчета

Оценка устроена как обычная проверка вопрос-ответ, а не как ручная экспертная разметка после каждого запуска.

Для baseline и generation-based sequential-оценки pipeline берет вопрос из eval-набора, превращает его в короткий QA-prompt и просит модель дать краткий ответ. Генерация идет детерминированно: без sampling, чтобы один и тот же запуск был максимально воспроизводимым.

После генерации ответ очищается от технического шума: служебных токенов, повторов роли, лишней пунктуации, повторенного текста вопроса и обрывков шаблона ответа. Затем очищенный ответ сравнивается с эталонным ответом и списком допустимых aliases. Если эталон или alias содержится в ответе после нормализации, вопрос считается правильным.

Дальше вопросы группируются по смыслу:

- прямые вопросы дают `reliability`;
- парафразы дают `generalization`;
- обратные вопросы дают `reverse`;
- соседние вопросы дают `neighbor`;
- locality-вопросы дают `fact_locality`;
- общий недоменный набор дает `global_locality`;
- общий нефтегазовый набор дает `domain_score`.

В single-edit часть post-edit оценки берется из внутреннего механизма `EasyEdit`: метод редактирует модель, затем `EasyEdit` проверяет подготовленные prompt-группы до и после правки и возвращает accuracy-подобные значения. Эти значения приводятся к единой схеме метрик, чтобы разные методы можно было сравнивать одной таблицей.

В sequential-edit после каждого шага дополнительно запускается generation-based проверка уже измененной модели. Для текущего факта считается качество текущей правки, а для всех ранее внесенных фактов считается, сохранились ли они после новой правки. Именно из этой проверки получается `retention`.

### Baseline metrics

Baseline считается до редактирования по трем наборам:

- `fact_questions`: вопросы вокруг 20 редактируемых фактов;
- `domain_questions`: общие нефтегазовые вопросы;
- `general_questions`: недоменные вопросы общего знания.

Эти метрики показывают, что модель знает до внесения правок.

### Single-edit metrics

В single-edit каждый факт редактируется независимо. Основные метрики:

- `reliability`: точность на прямом вопросе к отредактированному факту;
- `generalization`: точность на парафразах того же факта;
- `reverse`: точность на обратных вопросах;
- `neighbor`: сохранность близких доменных знаний;
- `fact_locality`: сохранность локальных вопросов, которые не должны измениться;
- `global_locality`: сохранность общих недоменных вопросов;
- `domain_score`: сохранность общего нефтегазового набора;
- `edit_quality`: сводная оценка одиночной правки.

`edit_quality` считается как гармоническое среднее из:

```text
reliability
generalization
mean(fact_locality, global_locality)
```

Гармоническое среднее выбрано специально: если один из важных компонентов близок к нулю, итоговая оценка тоже резко падает. Поэтому метод не может получить высокий `edit_quality`, если он хорошо записал факт, но разрушил locality.

### Sequential-edit metrics

В sequential-edit факты вносятся один за другим в одну изменяющуюся модель.

Основные метрики:

- `current_reliability`: точность на прямом вопросе текущего шага;
- `current_generalization`: точность на парафразах текущего шага;
- `retention`: сохранение direct-вопросов по всем уже внесенным фактам;
- `retained_generalization`: сохранение paraphrase-вопросов по ранее внесенным фактам;
- `global_locality`: точность на полном наборе general questions после текущего шага;
- `domain_score`: точность на полном наборе domain questions после текущего шага;
- `sequential_quality`: сводная оценка качества последовательного редактирования.

`retention` на шаге `N` считается по всем фактам, которые уже были внесены к этому шагу:

```text
retention(step N) = средняя direct accuracy по фактам 1..N
```

В полном прогоне `eval_mode=full`, поэтому retention считается по всем уже внесенным фактам, а не по подвыборке.

`sequential_quality` считается как гармоническое среднее из:

```text
current_reliability
current_generalization
retention
mean(current_fact_locality, global_locality)
```

Из-за этого метод получает низкий `sequential_quality`, если он хорошо редактирует текущий факт, но забывает предыдущие факты или портит общие знания.

### Replacement vs insertion

Для `LoRA`, `ROME` и `MEMIT` перед редактированием уточняется `target_old`.

- Если модель дает устойчивый корректный старый ответ, кейс считается ближе к `knowledge replacement`.
- Если ответ пустой, шумный или нестабильный, используется fallback к `target_new`, и кейс считается ближе к `knowledge insertion`.

Эта разметка важна для анализа: добавление неизвестного факта и замена уже известного знания могут иметь разные профили сложности.

## Single-edit результаты

| Метод | Edit quality | Краткий вывод |
|---|---:|---|
| `ROME` | `0.753` | лучший баланс качества и локальности |
| `WISE` | `0.738` | очень высокая reliability/locality, ниже generalization |
| `MEMIT` | `0.725` | сильный single-edit baseline, но хуже ROME |
| `LoRA` | `0.075` | хорошо пишет факт, но сильно портит locality |

В single-edit лучший метод по итоговой метрике — `ROME`.

## Sequential-edit результаты

| Метод | Sequential quality | Retention | Краткий вывод |
|---|---:|---:|---|
| `WISE` | `0.503` | `0.440` | лучший в последовательном режиме |
| `ROME` | `0.416` | `0.271` | сильный, но хуже удерживает цепочку |
| `MEMIT` | `0.073` | `0.063` | деградирует при накоплении правок |
| `LoRA` | `0.017` | `0.031` | почти не удерживает sequential-цепочку |

В sequential-edit лучший метод — `WISE`.

## Главные выводы

- Single-edit и sequential-edit дают разные ранжирования методов.
- `ROME` лучше всего подходит для одиночного редактирования в текущем benchmark.
- `WISE` лучше всего выдерживает последовательное накопление 20 правок.
- `LoRA` агрессивно меняет модель: целевой факт часто записывается, но locality и retention сильно страдают.
- `MEMIT` конкурентен в single-edit, но заметно деградирует в sequential.
- Большинство кейсов ближе к `knowledge insertion`, потому что модель не знает исходные нефтегазовые факты устойчиво.

## Pipeline

Основные команды:

```bash
make big-preflight
make big-smoke-pipeline
make big-run
make big-run-resume
make big-reports
```

Что они делают:

- `make big-preflight`: проверка CUDA, модели, данных и smoke-прогонов.
- `make big-smoke-pipeline`: маленький end-to-end запуск.
- `make big-run`: полный запуск baseline + single + sequential + reports.
- `make big-run-resume`: продолжить после обрыва.
- `make big-reports`: пересобрать отчеты из уже сохраненных raw-результатов.

## Ограничения

- Набор небольшой: 20 фактов.
- Оценка основана на exact/alias matching.
- Single-edit post-eval в основном использует EasyEdit internal metrics.
- Пока нет LLM-judge и более крупного отраслевого benchmark.
- Для выводов по `subject`, `relation`, `level` нужна дополнительная статистическая интерпретация.
